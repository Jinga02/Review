HTTP

  • HTTP(HyperText Transfer Protocol)는 W3 상에서 정보를 주고받을 수 있는 프로토콜이다.
  • HTML문서와 같은 resource(자원)들을 가져올 수 있도록 하는 프로토콜(규칙)
  • 클라이언트-서버 프로토콜이라고도 부르며, 클라이언트의 request(요청)과 서버의 response(응답)에 의해 통신된다.

  • HTTP의 특징

    1. stateless(무상태)
      • 동일한 연결에서 연속적으로 수행되는 두 요청 사이에 링크가 없다.
      • 즉, 응답을 마치고 연결을 끊는 순간 클라이언트와 서버 간의 통신이 끝나며 상태정보가 유지되지 않는다.
    2. connectionless(비연결성)
      • 요청을 한 후 응답을 받으면 바로 연결을 종료한다.
      • 즉, 불필요한 연결을 하지 않아 서버 유지 자원이 감소한다.

    HTTP Request Methods

    • resource(HTTP 요청의 대상)에 대한 행위(수행하고자 하는 동작)를 정의한다.
    • 즉, 리소스에 대해 수행 할 원하는 작업을 나타내는 메서드 모음을 정의한다.

    • 대표적인 HTTP Request Methods

      1. GET
        • 서버에 resource의 표현을 요청한다.
        • GET을 사용한응 요청은 데이터만 검색해야 한다.
      2. POST
        • 데이터를 지정된 resource에 제출한다.
        • 서버의 상태를 변경한다.
      3. PUT
        • 요청한 주소의 resource를 수정한다.
      4. DELETE
        • 지정된 resource를 삭제한다.

    HTTP response ststus codes

    • 특정 HTTP 요청이 성공적으로 완료되었는지에 대한 여부를 나타낸다.
      1. Informational responses(100-199)
      2. Successful responses(200-299)
      3. Redirection messages(300-399)
      4. Client error responses(400-499)
      5. Server error responses(500-599)

URI

  • Uniform Resource Identifier(통합 자원 식별자)
  • 인터넷에서 resource를 식별하는 문자열
  • 가장 일반적인 URI는 웹 주소로 알려진 URL이다. (자원의 위치로 자원을 식별)
  • 특정 이름공간에서 이름으로 resource를 식별하는 URI는 URN이다. (고유한 이름으로 자원을 식별)

    • URL

    • 네트워크 상에 리소스가 어디 있는지(주소)를 알려주기 위한 약속
    • URL은 다음과 같이 여러 부분으로 구성되며 일부는 필수이고 나머지는 선택사항이다.
    • URL의 구조
      1. scheme(or protocol)
        • 브라우저가 resource를 요청하는 데 사용해야 하는 프로토콜
        • URL의 첫 부분은 브라우저가 어떤 규약을 사용하는지를 나타난대.
      2. authority
        • scheme 다음으로 문자패터 ://으로 구분된 authority(권한)이 작성된다.
        • authority는 domain과 port를 모두 포함하며 둘은 :으로 구분된다.
        • domain name
          • 요청 중인 웹 서버를 나타낸다.
          • 어떤 웹 서버가 요구되는 지를 가리키며 직접 IP주소를 사용하는것도 가능하지만, 주로 domain name으로 사용한다.
          • 예를 들어 도메인 google.com의 ip주소는 142.251.42.142 이다.
        • port
          • 웹 서버의 리소스에 접근하는데 사용되는 기술적인 문(gate)이다.
          • HTTP 프로토콜의 표준 포트는 다음과 같고 생략이 가능하다.
          • HTTP - 80, HTTPS - 443
      3. path
        • 웹 서버의 리소스 경로이다.
        • 초기에는 실제 파일이 위치한 물리적인 위치를 나타냈지만. 오늘날에는 실제 위치가 아닌 추상화된 형태의 구조를 표현한다.
        • 예를 들어 /articles/create가 실제 articles폴더안에 create폴더 안을 나타내는 것이 아니다.
      4. parameters
        • 웹 서버에 제공하는 추가적인 데이터이다.
        • parameters는 & 기호로 구분되는 key-value 쌍 목록을 가진다.
        • 서버는 리소스를 응답하기 전에 이러한 parameters를 사용하여 추가 작업을 수행할 수 있다.
      5. anchor
        • resource의 다른 부분에 대한 앵커
        • 리소스 내부 일종의 "북마크"를 나타내며 브라우저에 해당 북마크 지점에 있는 콘텐츠를 표시한다.
        • 예를 들어 https://docs.djangoproject.com/en/3.2/intro/install/#quick-install-guide
        • 위의 요청에서 #뒤는 서버에 전달되지 않고 브라우저에게 해당 지점으로 이동할 수 있도록 한다.

    URN

    • Uniform Resource Name(통합 자원 이름)
    • URL과 달리 자원의 위치에 영향을 받지 않는 유일한 이름 역할을 한다.
    • URL의 단점을 극복하기 위해 등장했으며, 자원이 어디에 위치한지 여부와 관계없이 이름만으로 자원을 식별한다.
    • 하지만 이름만으로 실제 리소를 찾는 방법은 보편화 되어있지 않아 현재는 대부분 URL을 사용한다.

클라이언트 - 서버 구조

  • 클라이언트

    • 서비스를 요청하는 주체이다.
    • 웹 사용자의 인터넷에 연결된 장치
    • GoogleChrome 또는 Safari와 같은 웹 브라우저

  • 서버

  • 요청에 대한 서비스를 응답하는 주체이다.
  • 웹 페이지, 사이트 또는 앱을 저장하는 컴퓨터
  • 클라이언트가 웹 페이지에 접근하려고 할 때 서버에서 클라이언트 컴퓨터로 웹 페이지 데이터를 응답해 사용자의 웹 브라우저에 표시된다.
  • 상호작용 예시

      • 예를 들어, 우리가 google 홈페이지에 접속한다는 것은 무엇을 뜻하는지 알아보자.

    1. 인터넷에 연결된 전세계 어딘가에 있는 구글 컴퓨터에게 'google 홈페이지.html'파일을 요청한다.
    2. 구글 컴퓨터는 우리의 요청을 받고 'google 홈페이지.html'파일을 인터넷을 통해서 우리 컴퓨터에게 응답해준다.
    3. 전달받은 google 홈페이지.html파일을 웹 브라우저가 우리가 볼 수 있도록 해석해준다.

REST API

  • API

    • Application Programming Interface
    • 애플리케이션과 프로그래밍으로 소통하는 방법
      - 개발자가 복잡한 기능을 보다 쉽게 만들 수 있도록 프로그래밍 언어로 제공되는 구성
    • API를 제공하는 애플리케이션과 다른 소프트웨어 및 하드웨어 등의 것들 사이의 간단한계약이라고 볼 수 있다.
    • API는 복잡한 코드를 추상화하여 대신 사용할 수 있는 몇 가지 더 쉬운 구문으로 제공한다.

  • WEB API

    • 웹 서버 또는 웹 브라우저를 위한 API
    • HTTP 서비스이고 다양한 클라이언트에서 접근이 가능하다.

  • REST

    • Respresentational State Transfer
    • API Server를 개발하기 위한 일종의 소프트웨어 설계 방법론
    • 자원을 URI로 표시하고 해당 자원의 상태를 주고 받는 것을 의미한다.
    • REST의 구성요소

      • - 자원(Resource): URI
      - 행위(Verb): HTTP METHOD
      - 표현(Representations) : 자원과 행위를 통해 궁극적으로 표현되는 (추상화된)결과물

    • URI를 통해 자원을 표시하고 HTTP METHOD를 이용하여 해당 자원의 행위를 정해주며 그 결과를 받는 것을 말한다.

  • JSON

    • JavaScript 객체 리터럴, 배열, 스칼라 데이터를 표현하는 텍스트 기반의 방식
    • 속성-값 쌍, 배열 자료형 또는 기타 모든 시리얼화 가능한 값 또는 "키-값 쌍"으로 이루어진 데이터 오브젝트를 전달하기 위해 인간이 읽을 수 있는 텍스트를 사용하는 개방형 표준 포맷이다.
    • 비동기 브라우저/서버 통신 (AJAX)을 위해, 넓게는 XML(AJAX가 사용)을 대체하는 주요 데이터 포맷이다.
